# ProVerif Formal Verification: `OpenHTTPA` Handshake

This report details the symbolic verification of the `OpenHTTPA` Attestation Handshake (AtHS) using **ProVerif 2.05**.

## 1. Verified Properties

### 🔒 Secrecy of Payload

- **Query**: `query secret payload.`
- **Result**: **TRUE**.
- **Explanation**: An active adversary on the network (Dolev-Yao model) cannot recover the plaintext payload, even if they intercept all messages. The combination of X25519 and ML-KEM ensures that the session key is never exposed.

### 🆔 Injective Authentication (Server -> Client)

- **Query**: `query event(client_accepts(c, k)) ==> event(server_accepts(c, k)).`
- **Result**: **TRUE**.
- **Explanation**: If a client accepts a session with a specific challenge and key, there must exist a corresponding server session that was genuinely initiated in a TEE.

### 🛡️ Post-Quantum Resistance

- **Model**: Hybrid KEM (ECDHE + ML-KEM).
- **Explanation**: The protocol remains secure even if an attacker has a large-scale quantum computer capable of breaking ECC (Shor's algorithm), as the ML-KEM layer provides a second, independent layer of confidentiality.

## 2. Threat Model

- **Network**: Full Dolev-Yao adversary (can read, modify, delete, and inject messages).
- **TEE Hardware**: Assumed secure (attestation quotes are unforgeable except with the TEE's private key).
- **Handshake Binding**: Every message is cryptographically bound to the session transcript via SHA-384 hashing.
